# Универсальные правила разработки

Данный документ содержит набор правил и рекомендаций, обязательных к соблюдению при разработке проектов. Они обеспечивают единообразие кода, упрощают поддержку, масштабирование и командную работу.

---

## 1. Управление зависимостями

### 1.1. Проверка актуальности перед установкой
- **Правило:** Перед добавлением новой зависимости выполни проверку её актуальности, стабильности, размера и лицензии.
- **Действия:**
  - Используй `npm outdated` или `yarn outdated` для проверки уже установленных пакетов.
  - Для новых пакетов изучи:
    - количество загрузок (npm trends),
    - дату последнего обновления,
    - наличие открытых issues,
    - альтернативы с меньшим размером или лучшей поддержкой.
  - Предпочитай хорошо поддерживаемые и популярные библиотеки.
  - Фиксируй версии пакетов в `package.json` (без caret `^` для критических зависимостей) для воспроизводимости сборок.

## 2. Стилизация и темы

### 2.1. Использование Tailwind CSS + shadcn/ui
- **Правило:** Для стилизации используем Tailwind CSS в сочетании с библиотекой компонентов shadcn/ui.
- **shadcn/ui** — не пакет, а коллекция переиспользуемых компонентов, которые копируются в проект. Это даёт полный контроль над стилями.
- Все компоненты должны быть построены на основе утилит Tailwind.

### 2.2. Цветовая модель OKLCH и CSS-переменные
- **Правило:** Цвета задаются исключительно в цветовом пространстве OKLCH через CSS-переменные.
- **Почему:** OKLCH обеспечивает лучшую перцептивную равномерность и удобство ручного подбора оттенков.
- **Пример:**
  ```css
  :root {
    --primary: oklch(0.6 0.2 250);
    --background: oklch(1 0 0);
  }
  ```
- Все цвета темы должны определяться через CSS-переменные, которые переопределяются в зависимости от темы.

### 2.3. Светлая, тёмная и другие темы
- **Правило:** Проект должен с первого дня поддерживать светлую и тёмную темы, а также предусматривать возможность добавления других тем (например, контрастной, «детской» и т.д.).
- **Реализация:**
  - Используй атрибут `data-theme` или класс на корневом элементе.
  - Определи набор CSS-переменных для каждого селектора темы.
  - Для переключения тем используй JavaScript / контекст React.
  - Учитывай предпочтения системы через `prefers-color-scheme`.
  - Пример структуры:
    ```css
    :root,
    [data-theme="light"] {
      --bg: oklch(1 0 0);
      --text: oklch(0.2 0 0);
    }

    [data-theme="dark"] {
      --bg: oklch(0.2 0 0);
      --text: oklch(1 0 0);
    }

    [data-theme="high-contrast"] {
      --bg: oklch(1 0 0);
      --text: oklch(0 0 0);
    }
    ```

## 3. Правила именования

### 3.1. Переменные, функции, компоненты
- **Переменные:** `camelCase` (например, `userName`, `isLoading`).
- **Функции:** `camelCase`, глаголы (например, `fetchData`, `handleClick`).
- **Компоненты React:** `PascalCase` (например, `UserProfile`, `Button`).
- **Константы (глобальные):** `UPPER_SNAKE_CASE` (например, `API_BASE_URL`, `COLORS`).
- **CSS-классы (Tailwind):** только утилиты Tailwind, кастомные классы — исключительно в крайних случаях через `@apply` в CSS-файлах.

### 3.2. Файлы и папки
- **Файлы компонентов:** `PascalCase.tsx` (например, `UserProfile.tsx`).
- **Хуки:** `camelCase` с префиксом `use` (например, `useAuth.ts`).
- **Утилиты:** `camelCase` (например, `formatDate.ts`).
- **Стили:** если используется CSS-модули, то `[name].module.css`; глобальные стили — `globals.css`.

### 3.3. Переменные окружения
- **Формат:** `NEXT_PUBLIC_*` для публичных переменных (Next.js), `REACT_APP_*` для CRA, все заглавными буквами через подчёркивание.
- **Пример:** `NEXT_PUBLIC_API_URL`, `DATABASE_PASSWORD` (только на сервере).

## 4. Интернационализация (i18n)

### 4.1. Обязательное использование i18n
- **Правило:** Все тексты в интерфейсе должны быть вынесены в файлы локалей.
- **Инструмент:** Рекомендуется `next-i18next` (для Next.js), `react-i18next` или аналоги.
- **Структура:**
  ```
  public/locales/
    en/
      common.json
      home.json
    ru/
      common.json
      home.json
  ```
- **Ключи:** используй точечную нотацию (например, `home.welcome`).
- **Плюрализация и интерполяция:** должны поддерживаться библиотекой.

### 4.2. Автоматизация
- Настрой скрипты для извлечения ключей и синхронизации с переводческими сервисами (например, Lokalise, POEditor).

## 5. Контроль версий и качество кода

### 5.1. Автоматический коммит + Husky
- **Husky** используется для запуска хуков Git.
- **pre-commit:**
  - Запуск линтера (ESLint) с автоматическим исправлением (`--fix`).
  - Запуск форматтера (Prettier).
  - Проверка типов (TypeScript) — `tsc --noEmit`.
  - (Опционально) запуск单元 тестов, относящихся к изменённым файлам.
- **commit-msg:**
  - Проверка соответствия сообщения коммита стандарту [Conventional Commits](https://www.conventionalcommits.org/) (например, через `commitlint`).
- **pre-push:**
  - Полная сборка проекта.
  - Запуск всех тестов (unit, integration).

### 5.2. Автоматический коммит
- Не путать с автоматическим созданием коммитов. Имеется в виду, что процесс коммита автоматизирован (линтер, форматтер) и стандартизирован.
- Рекомендуется использовать `lint-staged` для проверки только staged файлов.

### 5.3. Дополнительные инструменты
- **ESLint** — обязательный набор правил (рекомендуется Airbnb или стандартный от Next.js).
- **Prettier** — единый стиль форматирования.
- **EditorConfig** — для согласованности настроек редактора.

## 6. Разработка: Mobile First

### 6.1. Принцип Mobile First
- **Правило:** Сначала проектируем и разрабатываем интерфейс для мобильных устройств (наименьший экран), затем с помощью медиазапросов добавляем стили для планшетов и десктопов.
- **Tailwind** поощряет этот подход: используй префиксы `sm:`, `md:`, `lg:` и т.д. для адаптации.
- **Пример:**
  ```jsx
  <div className="p-4 md:p-6 lg:p-8">
    {/* контент */}
  </div>
  ```

### 6.2. Тестирование на реальных устройствах
- Проверяй интерфейс на различных разрешениях в инструментах разработчика и, по возможности, на физических устройствах.

## 7. Дополнительные универсальные правила

### 7.1. Структура проекта
- **Feature-based** подход: группировка файлов по модулям/фичам, а не по ролям (components, hooks, utils).
- Пример:
  ```
  src/
    features/
      auth/
        components/
        hooks/
        api/
        types.ts
      profile/
        ...
    shared/
      ui/        # переиспользуемые UI-компоненты (кнопки, инпуты)
      lib/       # утилиты, конфиги
      types/     # глобальные типы
  ```

### 7.2. Компонентный подход
- Компоненты должны быть небольшими, переиспользуемыми, с чёткими интерфейсами (Props).
- Выделяй логику в кастомные хуки.
- Используй композицию вместо наследования.

### 7.3. Обработка ошибок
- Все асинхронные запросы должны обрабатывать ошибки и показывать понятные пользователю сообщения (через i18n).
- Используй границы ошибок (Error Boundaries) в React для отлова ошибок рендера.
- Настрой централизованный логгер (например, Sentry) для отслеживания ошибок в продакшене.

### 7.4. Доступность (A11y)
- Семантическая вёрстка (использовать правильные теги: `header`, `main`, `button` и т.д.).
- У всех интерактивных элементов должны быть доступные имена (aria-label, текст).
- Поддержка навигации с клавиатуры.
- Контрастность цветов должна соответствовать WCAG (проверять инструментами).

### 7.5. Производительность
- Ленивая загрузка компонентов (Next.js dynamic, React.lazy).
- Оптимизация изображений (next/image или аналоги).
- Минимизация перерисовок (React.memo, useMemo, useCallback — но с умом).
- Анализ бандла (next/bundle-analyzer).

### 7.6. Тестирование
- **Unit-тесты:** для утилит, хуков, сложной логики (Jest + React Testing Library).
- **Интеграционные тесты:** для критических пользовательских сценариев (можно использовать Cypress или Playwright).
- **Скриншотные тесты:** для визуального контроля (по желанию).

### 7.7. Документирование
- Ведение README.md с инструкцией по запуску, структуре проекта, переменным окружения.
- Документирование неочевидных решений в коде (комментарии) или в отдельной папке `/docs`.
- Storybook для UI-компонентов (опционально).

### 7.8. Безопасность
- Не хранить секреты в коде (использовать переменные окружения).
- Защита от XSS: экранирование вывода, использование доверенных библиотек.
- Проверка зависимостей на уязвимости (`npm audit`, `snyk`).

## 8. Заключение

Соблюдение этих правил обязательно для всех участников команды. Любые исключения должны обсуждаться и документироваться. Данные правила регулярно пересматриваются и дополняются по мере развития технологий и процессов.
